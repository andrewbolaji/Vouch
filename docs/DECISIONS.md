# Vouch -- Decisions Log

Architectural and product decisions with reasoning. Future-you reads this file and understands why the system is the way it is.

## Standing rules

| Date | Decision | Reason |
|------|----------|--------|
| 2026-05-07 | Em/en dash prohibition (standing rule) | No em dashes or en dashes anywhere in app, user-facing copy, or DECISIONS.md. AI tell. Use commas, periods, parentheses, or rewrite. Apply across all Blocks. |
| 2026-05-07 | 9th-grade reading level for all user-facing copy (standing rule) | All errors, empty states, confirmations, button labels, helper text readable by a 9th grader. No jargon. Concrete next steps in error messages. |
| 2026-05-07 | Tests as deliverables (standing rule) | A Block does not close with zero tests. Tests ship with the feature, not after. |
| 2026-05-07 | Repository pattern: presentation never imports Firebase directly (standing rule) | Screens and widgets never import FirebaseFirestore, FirebaseAuth, etc. All external service access goes through a service or repository layer. Keeps presentation testable and backend-swappable. |
| 2026-05-07 | Theme.of(context) + IconTheme for const-able color reads (standing rule) | Avoids passing runtime colors as constructor params (which breaks const). Widgets read colors from the inherited theme. Unlocks const widget constructors without sacrificing runtime theme switching. |
| 2026-05-07 | User guide (docs/USER_GUIDE.md) maintained live alongside code (standing rule) | Backfilled for Block 0. Updated every Block. Written for the actual user (9th-grade reading, no jargon, no plumbing words). Markdown working format, PDF/Word exported at v1 launch via pandoc. Taste frame (audience check) applied at every Block recap. Block does not close if shipped UI is missing from the guide. |

## Architectural decisions

| Date | Decision | Reason |
|------|----------|--------|
| 2026-04-27 | Flutter + Provider for initial state management | Fast cross-platform iteration. Provider is simple for the current scope (4 providers). Riverpod selected as the migration target once interaction tests provide a safety net. |
| 2026-04-27 | Riverpod selected over Provider continuation | Provider's BuildContext coupling makes testing harder and breaks in non-widget contexts. Riverpod eliminates this. Migration deferred until interaction tests pass (trigger now fired, tests complete). |
| 2026-04-27 | very_good_analysis over flutter_lints | Stricter rule set catches more issues at lint time. Enforces consistent style without manual review burden. Team-ready from day 1. |
| 2026-04-27 | cached_network_image over raw Image.network | Built-in disk + memory caching, placeholder/error widget support, avoids re-downloading on rebuild. Worth the dependency for an image-heavy app. |
| 2026-04-27 | Seed data with `kUseFirebase` flag, not Firebase from day 1 | Build the full UI and test suite before introducing network dependencies. Firebase wiring is a separate Block. Allows full offline development and fast test cycles. |
| 2026-04-27 | SharedPreferences for votes and saved restaurants (local persistence) | Near-zero failure rate on device storage. Acceptable for prototype phase. Migrates to Firestore when `kUseFirebase` flips. |
| 2026-04-27 | Three-tier membership: Free, Locals Pass, City Insider | Free gets Top 5 + vote/comment. Locals Pass ($4.99/mo) adds Top 10, saves, ad-free. City Insider ($9.99/mo) adds insider tips, badge, 2x vote weight. |
| 2026-04-27 | Client-side paywall gating only (for now) | All data is local seed data. No security risk in current state. Server-side gating is non-negotiable when Firebase is live. |
| 2026-04-27 | RevenueCat for IAP, mock mode until configured | Service layer stubbed with TODO comments. Real SDK activation requires App Store Connect products + RevenueCat dashboard setup. |
| 2026-04-27 | Mock auth with SharedPreferences persistence | Firebase Auth not yet configured. Mock stores email and display name only (no password, no token). No sensitive data at risk. |
| 2026-05-01 | Firestore region: us-central1 | Lowest latency to launch markets (Houston, NYC, LA, Chicago). Cannot change after project creation without full data migration to new project. Multi-region (nam5) rejected: adds cost, no benefit until international expansion. |
| 2026-05-01 | savedRestaurantIds as array on user doc, not subcollection | Single-read pattern. Atomic arrayUnion/arrayRemove. Won't hit 1MB even at 1000 saves (~10 bytes per ID). Subcollection would require a query per load. |
| 2026-05-01 | Comments as subcollection of restaurants, not array on doc | Unbounded growth. Requires paginated queries (page size 20, cursor-based with startAfter). Array on doc would hit 1MB limit. |
| 2026-05-01 | Vote doc ID = user UID (subcollection of restaurants) | Enforces one-vote-per-user at storage level. set() overwrites, not duplicates. No security rule needed to prevent duplication. voteCount denormalized on restaurant doc via Cloud Function onWrite trigger. |
| 2026-05-01 | Custom claims on auth token for membership gating | Security rules read claims from request.auth.token without extra Firestore reads. RevenueCat webhook sets the claim via Cloud Function. Claims limited to 1000 bytes but tier is a single string. |
| 2026-05-01 | Cloud Function proxy for insider field stripping | Firestore has no field-level read rules. getRestaurant Cloud Function checks custom claim and strips insiderTip/whatToOrder for non-entitled users. Direct Firestore reads for non-gated data only. |
| 2026-05-01 | Suggestion rate limit: counter doc + transaction, not Cloud Function gate | Faster (no HTTP call latency). Atomic read-check-write in standard Firestore transaction. Server-enforced. Client-side cap is UX only. |
| 2026-05-01 | MembershipProvider.initialTier constructor parameter for testability | Avoids mock purchase delays in tests. Defaults to MembershipTier.free. Zero production behavior change. Used by test helper buildTestApp. |
| 2026-05-01 | Comment ID via millisecondsSinceEpoch (seed data only) | Acceptable for local seed data. Firestore will use auto-generated IDs when wired. |
| 2026-05-07 | Framework files stored in docs/ | AI_BUILD_FRAMEWORK.md, BLOCK_TEMPLATES.md, REVIEW_FRAMEWORK.md, DECISIONS_AND_WISHLIST_DISCIPLINE.md organized under docs/ for separation from source. |
| 2026-05-07 | Concurrent sessions: allow | Standard consumer-app expectation. Firebase handles natively. No work needed. |
| 2026-05-07 | Password reset: defer to Firebase default web handler for v1 | In-app deep link handling requires go_router, which is itself deferred. Pinned in WISHLIST.md v1.1 with trigger: "When go_router ships." |
| 2026-05-07 | Account deletion: v1 scope, non-negotiable | Apple App Store guideline 5.1.1(v) requires it. Block 1 ships auth-level deletion (FirebaseAuth.currentUser.delete()). Cloud Function for Firestore user data cleanup ships in Block 2 or 3 when Firestore data exists. Profile menu entry with confirmation modal. |
| 2026-05-07 | AuthService owns authStateChanges() stream | Enforces single subscription per app instance. Preserves repository pattern (callers never import FirebaseAuth). Automatic disposal when MultiProvider tears down. Implementation: late final StreamSubscription in constructor, cancelled in dispose(). |
| 2026-05-07 | Auth methods for v1: email + password, Google, Apple | Email link/passwordless deliberately deferred. Google/Apple already solve the password-averse use case. Email link requires deep-link infrastructure deferred to go_router. Pinned in WISHLIST.md v2 with trigger: "go_router ships AND password-fatigue feedback." |
| 2026-05-09 | Suggestion daily cap: 1/day (was 3/day) | 1 is enough to signal engagement without flooding the queue before moderation exists. 24-hour reset window unchanged. |
| 2026-05-09 | Premium vote weight: 3x | 2x is mathematically too small to feel meaningful or justify $5/mo; 5x feels oligarchic, risks App Store "pay-to-win" framing, and silences free users to the point of breaking the brand promise. 3x is meaningful but coordinated free-user communities can still shift rankings. "Triple" is also the most legible language for users. |
| 2026-05-09 | No ads in v1 | Ad-free is the foundational premium feature; introducing ads to free tier without a meaningful free user base destroys the premium pitch and generates near-zero revenue. Trigger to revisit: 10K+ DAU AND ad-free is no longer the strongest premium differentiator (e.g., after vote weight has been repositioned as the headline premium feature). |
| 2026-05-09 | Vote breakdown display visible to all users including free tier | Transparency is the ethical defense for the premium-vote-weight model. Users see how rankings form and decide for themselves if they trust the math. Paywalling the receipts would defeat the point. |
| 2026-05-09 | Launch city scope expanded to 5 cities (Houston, Atlanta, New Orleans, Chicago, San Francisco) | Curated via TikTok food creator candidates + Reddit + Eater + Michelin pipeline. Andrew sources Top 10 candidates per city; voting forms the final Top 5 visible to free users with bottom 5 dimmed/locked behind premium. Seed data city count unchanged in this Block. NYC deferred: too large and too competitive at launch; revisit after 3 months live with the initial 5. |
| 2026-05-09 | App bundle ID: com.example.majorcitymusteats | Matches the legacy Firebase Project ID kept as project lore. Don't rename without coordinated Firebase reconfigure. All platforms (iOS, Android, macOS, Linux, web) aligned to this ID. |
| 2026-05-09 | Google Places usage policy: intake-only | Vouch uses Google Places API only at suggestion intake (autocomplete + one Place Details call per accepted suggestion). All restaurant data subsequently lives in Vouch's Firestore. Vouch does NOT re-fetch from Google on page loads. Reasoning: re-fetching on every page view would cost ~$2,500/month at 1K DAU; current architecture caps spend at ~$50-100/month at the same scale. Place ID stored as the canonical dedup key. |
| 2026-05-26 | Accent color: #FF5436 (vermilion). Supersedes #FF3B5C (pink, Block 2) | On the new warm near-black background (#0F0D0B) the pink read cool-on-warm (clashing, social-media-coded); vermilion reads warm-on-warm (integrated, appetite-forward, editorial). Vermilion scored highest contrast of the candidates (6.07:1 on background, 5.59:1 on surface, both AA-passing). Makes the rank badges read as an award/seal rather than an alert. Amber (#E8945A) was rejected (overused across other projects). This is the locked accent. |
| 2026-05-26 | Insider content gating: insiderNotes subcollection, not Cloud Function proxy. Supersedes FIRESTORE_SCHEMA.md recommendation. | The schema doc recommended a Cloud Function proxy that reads the restaurant doc and strips insiderTip/whatToOrder for non-entitled users. This fails open: if the function errors (cold start timeout, quota, runtime exception), insider fields are exposed on the restaurant doc via direct Firestore reads. The subcollection approach (/restaurants/{id}/insiderNotes/{noteId}) enforces the boundary at the storage layer with a security rule requiring cityInsider custom claim. Zero leak failure mode. Also eliminates a Cloud Function from the hot read path (lower latency, lower cost). |
| 2026-05-26 | Suggestion rate limit: server-side Cloud Function, not client transaction with date-key validation | The schema doc proposed a client-side Firestore transaction with a date-keyed counter doc, but the date key is client-supplied and could be backdated to dodge the cap. A Cloud Function owns the date key (uses server time), making the cap un-spoofable. The client calls the function; the function reads the counter, checks the cap, writes the suggestion and increments the counter atomically. Rules still restrict direct suggestion writes to require auth, but the function is the enforced path. |
| 2026-05-26 | Vote aggregation: incremental FieldValue.increment, not re-count | The Cloud Function on vote create/delete uses FieldValue.increment(1) and FieldValue.increment(-1) respectively, not a re-count of the entire votes subcollection. A re-count would read N docs per vote at scale (e.g., 5,000 reads for a 5,000-vote restaurant), which is both slow and expensive. Incremental is O(1). |
| 2026-06-07 | Saves are server-authoritative on sign-in (replace, not merge) | On sign-in, SavedProvider fetches the full saved list from Firestore and replaces local state entirely. Alternative was merge (union of local + server), but merge risks resurrecting IDs the user deleted on another device. Server is the single source of truth. Local cache exists only as a same-uid offline fallback. |
| 2026-06-07 | Per-uid SharedPreferences cache: same-uid offline fallback only | Cache key is saved_restaurant_ids_$uid (and suggestion_remaining_$uid). Written after every successful server load or toggle. Read ONLY when the server load fails for the SAME uid. Never read cross-user, never merged. On sign-out, state is cleared in memory (cache on disk is inert because it requires the matching uid to be loaded). Alternative was no cache (simpler), but offline-first UX on flaky networks is worth the 6 lines. |
| 2026-06-07 | Providers react to auth via internal AuthService listener, not ProxyProvider | SavedProvider and SuggestionProvider subscribe to AuthService.addListener in their constructors, with removeListener in dispose. Alternative was ChangeNotifierProxyProvider in main.dart, but that creates a second reaction path (provider update callback + internal listener), which doubles auth-change handling. Single internal listener is simpler, testable without widget tree, and keeps provider self-contained. |
| 2026-06-08 | Unranked restaurants use rank 9999 sentinel (kUnrankedRank), not 0 | Firestore orderBy('rank') naturally places 9999 after 1-10 without client-side sort. UI suppresses rank badge and vote count at these values. The sentinel is a named constant referenced everywhere (no bare 9999 literals). Compatible with the future rank-from-votes Cloud Function, which will overwrite the sentinel when it ranks. |
| 2026-06-08 | displayOrder from CSV row order for day-1 browse | Editorial curation determines initial browse order. Voting reshapes ranking over time. displayOrder is static and never recalculated. Secondary sort in Firestore query: orderBy('rank').orderBy('displayOrder'). |
| 2026-06-08 | Doc ID = hou-{placeId} for Houston restaurants | Place ID is globally unique and deterministic. Dedup key IS the doc ID. No slug collision risk. Demo docs (hou-1 through hou-10) use different ID scheme and are cleaned via --clean-demo before real seed. |
| 2026-06-08 | Demo data removed before real seed via --clean-demo | Houston-only launch. NYC/LA/Chicago demo docs removed from prod. Prevents duplicate and mixed-quality data. In-app SeedData fallback is read-only, never writes to Firestore. |
| 2026-06-08 | isMobileVenue stored, UI deferred | Flag written to Firestore. Food trucks show home-base address until map/location screen ships. Trigger: map/location detail screen. |
| 2026-06-08 | Places match requires name similarity (>= 0.5 token overlap) + Houston metro city/bbox match | Wrong Place ID is worse than no data. Address-less rows always go to manual review. Metro check covers suburbs (Katy, Pearland, Spring, Sugar Land, etc.) via city name set and lat/lng bounding box. |
| 2026-06-08 | openingHours stored but not surfaced | Stale hours are worse than no hours. Stored for the v2 refresh job. |
| 2026-06-08 | No rank-from-votes process exists yet | Vote Cloud Functions increment/decrement voteCount but nothing computes rank from voteCount. Top 10 stays empty until the rank computation Cloud Function ships (next Block). |
| 2026-06-08 | Demo image override layer (display-time only, never touches Firestore) | PRE-LAUNCH CHECKLIST: kUseDemoImageOverrides must be set to false or the file deleted before any public store build. Removal: delete lib/config/demo_image_overrides.dart, assets/demo/, and the pubspec assets entry. |
| 2026-06-09 | Comment moderation: report + block, manual review only (App Store Guideline 1.2) | Reports queue in Firestore for admin console review. No auto-hide, no auto-removal. Block is client-side filtering on blockedUserIds array. |
| 2026-06-09 | Report rate limit is client-side only (5/day), not server-enforced | Unlike suggestions (which use a Cloud Function for tamper-proof server-side cap), the report rate limit is enforced by the client checking reportCounts before submitting. The Firestore security rules validate report field types and require reporterUid == auth.uid, but do NOT check the count before allowing a create. Acceptable because: (1) a report is a low-value write (it queues for manual review, not an action), (2) a determined attacker spamming reports is visible in the admin queue, and (3) adding a Cloud Function for reports adds cost and latency for a low-priority abuse vector. If report abuse becomes real, add a CF gate matching the suggestion pattern. |
| 2026-06-09 | Block filtering is client-side, not Firestore query | Firestore cannot efficiently exclude a dynamic list of user IDs in a query. The blockedUserIds list is loaded on sign-in and comments are filtered before rendering. Acceptable at current scale (blocklists are small). |
| 2026-06-09 | Account deletion (App Store Guideline 5.1.1): comments anonymized, votes deleted | Comments by deleted users are anonymized (userId -> "deleted", userName -> "Deleted user", isInsider -> false) to preserve reply threads. Votes are deleted; in production the Firestore onVoteDeleted trigger fires and calls applyVoteDeleted to decrement aggregate voteCount (trigger wiring is Firebase infrastructure, not tested by our suite). The cleanup logic lives in a shared deleteUserData function (functions/src/user_cleanup.ts) called by both onUserDeleted and the tests, same pattern as filterBlockedComments. Vote logic lives in applyVoteCreated/applyVoteDeleted (functions/src/vote_aggregation.ts), called by both the triggers and the tests. No reimplementation in tests. Mechanism: client calls FirebaseAuth.delete(), which fires the onUserDeleted auth trigger. Re-auth: requires-recent-login is caught and surfaced as a re-auth dialog (password for email, provider re-auth for Google/Apple). Retry is capped at one attempt. The uid is captured before deletion for clearing uid-scoped SharedPreferences keys. |
| 2026-06-11 | Ranking engine: time-decayed score, daily scheduled recompute | Score = sum of weight * 2^(-ageDays / 90). Half-life 90 days (tunable constant). Scheduled Cloud Function at 06:00 UTC daily reads all vote docs per restaurant, computes score, assigns contiguous ranks 1..N per city. Pure math lives in rank_engine.ts (no Firestore), orchestrator in rank_recompute.ts. Client reads rank field as before, no sort-path changes. |
| 2026-06-11 | Seed votes are real vote documents, not synthetic voteCount inflation | The ranking engine reads only the /votes subcollection (with timestamps for decay). Seed data creates 15-30 real vote docs per restaurant with timestamps spread over 0-60 days. The voteCount field on the restaurant doc is the cosmetic display number (left at the large seeded values), not used by the engine. Seed votes decay naturally over months, transitioning the list from editorial to community-driven. |
| 2026-06-11 | Voting requires sign-in (fixes anonymous-vote bug) | The toggleVote call previously hardcoded userId='anonymous', meaning all votes overwrote the same doc. Fixed: userId is the real UID. Tapping vote while signed out opens the sign-in sheet. One-vote-per-user is structurally guaranteed by doc ID = UID. |
| 2026-06-11 | Vote weight field defaults to 1, enforced by security rules | Vote docs now include weight: 1. Security rules enforce weight == 1 on create. The ranking engine reads weight in its decay formula. When verified-visit detection ships, update the rule to allow weight > 1 with a verification claim. |
| 2026-06-11 | onUserDeleted is Gen1; Node runtime pinned to 22 | The after-delete auth trigger (auth.user().onDelete) is Gen1-only; there is no 2nd gen equivalent. Gen1 does not support Node 24. The functions runtime is set to Node 22 (supported on both Gen1 and Gen2) so all five functions deploy on the same runtime. Do not bump to Node 24 without first migrating onUserDeleted to a Gen2 mechanism. |
| 2026-06-11 | Switch to incremental decayed-score counter when vote volume is large | Current approach reads all vote docs per restaurant at recompute time. Trigger: when any single restaurant crosses 500 real vote docs. At that point, switch to an incremental model where onVoteCreated/onVoteDeleted maintain a decayedScore counter directly (adding/removing the vote's decayed contribution), and the daily function only recalculates the time-decay adjustment. This avoids O(N) reads per restaurant per recompute. |
| 2026-06-14 | iOS bundle ID: com.thunderrivertech.vouch (final) | com.vouch.app could not be registered under the team, so the final iOS bundle ID is com.thunderrivertech.vouch. A new iOS app was registered in the Firebase project (majorcitymusteats) for this bundle ID, and a new GoogleService-Info.plist was downloaded. Android, macOS, and Linux still carry the placeholder com.example.majorcitymusteats and will be aligned when those platforms ship. Supersedes the com.vouch.app entry and the iOS portion of the 2026-05-09 bundle ID decision. |
| 2026-06-09 | When seed or demo data changes, grep tests for hardcoded data values, not just names (standing rule) | A findsNothing assertion against a string you deleted passes for the wrong reason. Caught: "2-hour weekend wait" (Turkey Leg Hut's old tip) was still asserted as findsNothing after the restaurant was replaced, making the security test vacuous. Also caught: Himalaya and Xochi (old rank 6-7) in a paywall test, both absent from seed, so the paywall assertion could never fail. Fix: canary strings must exist in the current seed data so findsNothing proves the gate held, not that the string was garbage-collected. |
